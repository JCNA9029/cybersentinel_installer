CLI
Interactive Console — CyberSentinel.py
Full-featured 16-item terminal menu. Run as Administrator. All scan tiers available interactively.
python CyberSentinel.py
Menu 1
Scan Local File or Directory
Routes a file through the full 4-tier pipeline: Allowlist → SQLite cache → Cloud consensus (VT, OTX, MetaDefender, MalwareBazaar) → LightGBM ML → LLM triage report. Batch-scans entire directories for .exe, .dll, .sys, and more.
Scenarios
Analyst receives a suspicious installer from an email attachment and wants a quick verdict before opening.
SOC sweep of a shared network folder after a phishing campaign, batch-scanning all dropped executables at once.
Validating a software delivery from a vendor by scanning the full installer directory for embedded malicious DLLs.
Menu 2
Scan Hash / IP / URL / IoC Batch
Looks up SHA-256 hashes, IP addresses, or URLs directly against cloud threat intel APIs without needing the original file. Accepts a .txt file of indicators for bulk triage.
Scenarios
IR team receives a list of IoCs from threat sharing (ISAC) and needs rapid bulk triage against live APIs.
Checking a suspicious outbound IP seen in firewall logs for known C2 association in OTX and VirusTotal.
Validating hashes from a vendor's SBOM against MalwareBazaar before deploying a software update.
Menu 3
Live EDR — Analyze Active Memory
Enumerates running processes, resolves PIDs to executable paths, and routes the selected process through the full scan pipeline — effectively scanning a live process on disk.
Scenarios
Investigating a high-CPU process on a production server whose binary name looks legitimate but path is unusual.
Post-incident triage: a process spawned by a phishing macro is still alive and an analyst wants a verdict without killing it first.
Routine health check — scanning all running processes on a new machine before adding it to a fleet.

Menu 4
LoLBin Abuse Checker
Five-layer static analysis of a process name + command line for Living-off-the-Land Binary abuse. Strips obfuscation (carets, env vars) before matching 22 built-in patterns plus the live LOLBAS feed. Flags high-entropy arguments.
Scenarios
SOC analyst suspects certutil was used to download a payload; pastes the observed command line for instant classification.
Hunting for mshta- or regsvr32-based execution in a batch of forensic process logs collected from an endpoint.
Red team operator verifying whether a chosen LOLBin + argument combination would be flagged by the detector before executing.

Menu 5
BYOVD Vulnerable Driver Scan
Scans System32\drivers for drivers whose SHA-256 matches the LOLDrivers vulnerable driver database. Falls back to filename matching when hashes are absent. Reports CVEs and known abuse techniques.
Scenarios
Hardening sweep of a new Windows image before mass deployment — identifying any bundled vulnerable drivers.
Post-breach review: checking whether attackers dropped a signed vulnerable driver to disable EDR (BYOVD technique).
Compliance audit requiring documentation that no known-vulnerable kernel drivers are present on regulated endpoints.
T1068T1014
Menu 6
Attack Chain Correlation
Queries the shared event_timeline for completed multi-stage attack chain matches across a 10-minute correlation window. Displays any active chains with their MITRE stage breakdown and L1 triage steps.
Scenarios
Alert fatigue reduction: analyst reviews all open chains after an alert storm to prioritize which individual alerts form a coherent kill chain.
During an active incident, checking whether the observed PowerShell execution and outbound C2 connection have been correlated as a "Fileless → C2" chain.
End-of-shift review to confirm no silent attack chains completed overnight while the team was offline.

Menu 7
Baseline Environment Manager
Starts or stops behavioral profiling (LEARN mode) and shows statistics. In LEARN mode the engine silently records every process and its network destinations. In DETECT mode unknown processes receive a risk penalty fed into the Dynamic Risk Score.
Scenarios
Establishing a 24-hour behavioral baseline on a new air-gapped industrial control system before enabling live detection.
After a software update, re-baselining an endpoint so that the new processes added by the update are no longer flagged as unknown.
Checking baseline stats mid-investigation to see how many processes running right now are absent from the known-good profile.
Menu 8
Fileless / AMSI Alert History
Displays all alerts stored in the fileless_alerts table — anonymous RWX memory detections, AMSI ScriptBlock hits, and in-memory script analysis results — with timestamps, process context, and matched patterns.
Scenarios
Analyst reviewing overnight PowerShell alert history to identify which scripts triggered AMSI pattern matches and their originating process.
Forensic review after a suspected fileless attack: confirming that RWX memory regions were detected and at what time relative to other events.
Compliance reporting: exporting evidence that in-memory attack vectors are actively monitored on a regulated endpoint.

Menu 9
BYOVD Real-Time Kernel Monitor
Starts an interactive WMI-backed background monitor that polls Win32_SystemDriver every 10 seconds. New driver loads are checked against LOLDrivers in real time and immediately trigger console alerts and webhook notifications.
Scenarios
During a live red team exercise, a blue team operator runs this monitor to catch any BYOVD driver drop in real time.
Monitoring a high-value server during a patch window when unauthorized driver loads could indicate compromise.
Incident response: keeping an interactive session open on a compromised host to catch any new driver implant attempts while remediation is in progress.

Menu 10
AMSI Hook — Script & Memory Scan
Two sub-modes: (a) scan a script file against 20+ AMSI bypass and credential theft patterns with MITRE technique mapping; (b) scan a running process PID for anonymous RWX memory regions via VirtualQueryEx — flagging shellcode staging.
Scenarios
Analyst receives a suspicious .ps1 script from an email and wants to assess its obfuscation level before executing in a sandbox.
Hunting for in-memory shellcode: scanning a suspicious process PID for RWX regions after detecting anomalous behavior via EDR telemetry.
Malware reverse engineering support: using the script scanner to map which MITRE techniques a captured script implements before writing a detection rule.

Menu 11
Network Containment Control
Toggles Windows Firewall to blockinbound/blockoutbound on all profiles (isolate) or restores to default (restore). Uses netsh advfirewall with no terminal flash. Requires Administrator. Instant host quarantine without physical access.
Scenarios
Active ransomware detected mid-encryption: immediately isolating the host's network to prevent lateral movement and C2 communication while preserving the OS for forensic imaging.
Coordinated response across a SOC playbook — an automated or manual step that isolates an infected endpoint the moment a CRITICAL verdict is confirmed.
Controlled test of network isolation as part of an IR drill, then restoring connectivity to verify the restore path works correctly.
Menu 12
Update Threat Intelligence Feeds
Pulls all four open-source feeds — LOLBAS, LOLDrivers, Abuse.ch JA3 fingerprints, Feodo Tracker C2 IPs — and validates each before overwriting the cached copy. Stores update timestamps in update_meta.json.
Scenarios
Start-of-week SOC routine: refreshing all threat intel feeds before beginning the week's hunting activities.
After a major threat actor disclosure (new LOLBin abuse, new C2 infrastructure), immediately pulling updated feeds to ensure detections are current.
Before disconnecting a laptop for an air-gapped deployment, ensuring all feeds are up to date so the offline fallback data is as fresh as possible.
Menu 13
Configure API Keys & Settings
Encrypted (hardware-bound AES-128) read/write of VirusTotal, AlienVault OTX, MetaDefender, and MalwareBazaar API keys, the SOC webhook URL, and the LLM model name. Keys cannot be decrypted on a different machine.
Scenarios
Initial deployment: entering API keys for the first time during tool setup on a new SOC workstation.
Rotating API keys after a security review, updating all four cloud service credentials without touching any config file directly.
Connecting a Discord/Slack webhook so that all future CRITICAL verdicts and attack chain alerts are automatically posted to the SOC channel.
Menu 14
View Threat Cache
Renders the SQLite scan_cache table — SHA-256, filename, verdict, and timestamp for every previously scanned artifact. Enables instant repeat-detection lookups without re-querying cloud APIs. Useful for audit trails.
Scenarios
Producing a quick audit log of all files scanned during a week-long IR engagement to hand off to the client.
Verifying that a file previously marked MALICIOUS is still in the cache so it won't bypass future scans via the Tier 0.5 shortcut.
Cross-checking a new IoC hash against the local cache before spending an API quota call on a cloud lookup.
Menu 15
View Analyst Feedback History
Displays the full correction queue with status lifecycle (PENDING_REVIEW → PENDING → TRAINED / REVOKED / CONFLICTED), timestamps, and the associated SHA-256. Used to audit and manage the adaptive learning pipeline.
Scenarios
Checking how many false positive corrections are queued as PENDING before triggering a scheduled retraining session.
Auditing a CONFLICTED correction where two analysts labeled the same hash differently, to decide which label to approve manually.
Verifying that a model rollback successfully moved a TRAINED correction back to REVOKED after a retraining session introduced regressions.
Menu 16
Generate Session Report & Exit
Compiles all scan results, verdicts, SHAP explanations, risk scores, and alerts from the current session into a structured report saved to the Analysis Files directory, then exits the console cleanly.
Scenarios
End of an IR engagement: generating a forensic session summary to include in the client-facing incident report.
Shift handoff in a 24/7 SOC — outgoing analyst generates a report so the incoming analyst has full context on what was scanned and flagged.
Archiving a completed threat hunting session's findings before closing the tool, ensuring all AI triage notes and MITRE mappings are preserved.

GUI
Desktop Application — gui.py (PyQt6)
15-page threaded desktop app. All scan operations run in QThread workers — the UI never blocks. Replaces CLI prompts with Qt dialogs for quarantine, isolation, and AI report display.
python gui.py
Page 1
Dashboard Overview
Top-level summary of all active detections, recent scan verdicts, attack chain status, drift alerts, and risk score distribution. Entry point for every shift and the first screen seen after launch.
Scenarios
Morning briefing: SOC analyst opens CyberSentinel and scans the dashboard for any overnight CRITICAL alerts or completed attack chains before starting triage.
Management demo: showing a non-technical stakeholder the live threat posture of an endpoint in an easily digestible visual format.
Page 2
Scan File
Drag-and-drop or Browse file picker to route a single file or directory through all four tiers. Output renders in a color-coded ConsoleWidget. Quarantine authorization is a Qt dialog — no terminal interaction needed.
Scenarios
Level 1 analyst with no CLI experience scanning a suspicious attachment submitted via the helpdesk ticketing system.
Scanning a USB drive dropped in the parking lot (classic physical social engineering scenario) by dragging its root folder into the scan window.
Page 3
Scan Hash / IoC
Text input or file picker for SHA-256 hashes, IPs, URLs, and batch .txt IoC lists. Results table shows per-API verdict, engine count, and Dynamic Risk Score. No file required — cloud-only lookup path.
Scenarios
Pasting a hash from a SIEM alert directly into the GUI to get a multi-source verdict without leaving the tool.
Loading a bulk IoC list from an ISAC report and triaging all 200 indicators in a single batch while reviewing results in the table.
Page 4
Live EDR
Live process list with PID, name, and path. Click a process to scan its binary through all tiers. SHAP explanations and LLM triage report render inline. Designed for real-time process investigation without a terminal.
Scenarios
Non-CLI operator investigating a high-memory process flagged by task manager, clicking it in the list to get a full scan verdict and AI explanation.
Quickly scanning all processes associated with a newly logged-in user account to detect any malicious persistence mechanisms they brought with them.
Page 5
LoLBin Abuse
GUI frontend for the 5-layer LOLBAS detection engine. Input a process name and command line, or load from a process list. Displays matched patterns, Shannon entropy score, and MITRE technique tags inline.
Scenarios
Blue teamer reviewing process creation events exported from a SIEM, pasting command lines one by one to identify which ones are LOLBin abuse.
Threat hunter analyzing a set of mshta and certutil command lines from endpoint telemetry to determine which are malicious vs. legitimate admin activity.
Page 6
BYOVD Drivers
Runs the BYOVD scan against System32\drivers and displays a table of all detected vulnerable drivers with their CVE references, known abuse techniques, and LOLDrivers database match confidence.
Scenarios
Pre-deployment hardening review: a system administrator scanning a gold image for bundled vulnerable drivers before mass rollout.
Post-breach investigation: quickly checking whether an attacker planted a known BYOVD driver to blind the endpoint protection platform.
Page 7
Attack Chains
Visualizes completed and in-progress attack chains from the chain_alerts table. Each chain shows its stages, timestamps, involved processes, severity, and MITRE technique links. Webhook send status is shown per chain.
Scenarios
Incident commander reviewing which kill chains have completed during an active breach to assess attacker stage progression.
Threat hunting: confirming a DGA Beacon → C2 Resolve chain pattern matches the TTPs of a known APT group being tracked.
Page 8
Baseline
Start/Stop behavioral learning, view baseline stats (profile count, last updated, unknown process rate), and inspect the per-process behavioral profile table. Controls the risk penalty input to the Dynamic Risk Scorer.
Scenarios
Setting up CyberSentinel on an OT/ICS system where the process inventory is known and static — baselining for 24 hours then enabling detect mode for permanent monitoring.
After a legitimate software update, reviewing the unknown process rate spike and selectively approving new processes into the baseline without full re-learning.
Page 9
Fileless / AMSI
Table view of all fileless_alerts: RWX memory hits, AMSI ScriptBlock pattern matches, and script file scan results. Each row shows process, timestamp, matched indicators, and MITRE technique tags. Refresh button polls latest events.
Scenarios
Reviewing all PowerShell ScriptBlock alerts from an endpoint that triggered multiple AMSI events overnight to identify which script was the initial stager.
Forensic analysis: correlating the timestamp of an RWX memory detection with C2 connection alerts to reconstruct the shellcode injection sequence.

Network
Displays all C2 alerts (Feodo IP hits, DGA domain detections, JA3 fingerprint matches) and provides Isolate / Restore network buttons backed by Windows Firewall. Confirmation dialog prevents accidental isolation.
Scenarios
Analyst sees a Feodo C2 IP hit and immediately clicks Isolate to cut the endpoint off the network while escalating to the IR team.
Reviewing JA3 TLS fingerprint matches to identify outbound C2 traffic that is otherwise encrypted and invisible to payload inspection.

Intel Feeds
Shows last-update timestamps for all four feeds (LOLBAS, LOLDrivers, JA3, Feodo), feed sizes, and a Refresh All button. Validates each feed before writing, rejecting empty or malformed responses.
Scenarios
Verifying that threat feeds were successfully refreshed after a new threat actor campaign was disclosed and checking that LOLDrivers now includes the newly published vulnerable driver.
Diagnosing a stale feed warning by checking the update timestamp and manually triggering a refresh when automated updates failed.

Settings
GUI form for entering and saving API keys, webhook URL, and LLM model. All values are encrypted with the hardware-bound Fernet key on save. Includes a webhook test button that sends a sample payload.

Full correction queue management: view all FP/FN submissions with lifecycle status, approve or reject CONFLICTED entries, revoke PENDING corrections, and manually trigger retraining. Conflict resolution dialog for contradictory labels.

Daemon
Headless Monitor — --daemon PATH
Seven concurrent background threads covering files, processes, network, TLS, PowerShell, and attack chain correlation. No user interaction required after startup.
python CyberSentinel.py --daemon "C:\Watch\Folder"
Thread 1
Watchdog
Folder File Monitor
Intercepts new file drops matching watched extensions (.exe .dll .sys .apk .elf .pdf .bat .ps1 .vbs .hta) in the configured folder. A 2-second sleep absorbs write completion before routing the file through the full scan pipeline.
Monitoring a shared network drop-zone where all inbound files from external partners land — automatically scanning every new file before it can be accessed by internal users.
Watching an email gateway's attachment extraction directory so every attachment is scanned at the perimeter before delivery to the mailbox.
Thread 2
WMI
WMI Process & Driver Hook
Fires on every new Win32_Process and Win32_SystemDriver creation event. Routes each event through the LoLBin detector, BYOVD detector, and baseline engine in real time. Zero polling delay — event-driven via WMI subscription.
Detecting certutil spawned by a Word macro within milliseconds of execution, correlating parent/child process lineage to confirm LOLBin abuse.
Catching a vulnerable driver dropped and loaded by ransomware during its EDR-blinding phase before it can disable protection.
T1218T1068T1059
Thread 3
ETW
Event ID 4688 Process Creation Monitor
Reads Windows Security Event Log for process creation events (Event ID 4688). Provides a secondary, ETW-backed detection path for process launches that complements the WMI hook with audit log evidence.
Providing a forensic-quality record of every process launch with full command line (when Windows audit policy is configured), suitable for SIEM forwarding.
Detecting process creation that bypasses WMI (e.g., via direct syscalls) by relying on the kernel-level audit event instead.
Thread 4
Feodo
C2 IP Connection Monitor
Continuously polls active network connections via psutil and cross-references remote IPs against the Feodo Tracker C2 botnet blocklist. Fires a webhook alert on any match and writes to c2_alerts.
Detecting an infected host beaconing to a known Emotet or Qakbot C2 IP the moment the connection is established, before any payload is delivered.
Monitoring an already-compromised host under controlled conditions to understand C2 communication patterns and timing without cutting network access immediately.
T1071T1041
Thread 5
JA3
Passive TLS JA3 Fingerprint Sniffer
Captures TLS ClientHello packets passively via Scapy + Npcap, computes JA3 fingerprint hashes, and cross-references against Abuse.ch SSLBL. No traffic decryption required. Requires Npcap installed.
Identifying C2 traffic from malware that uses a distinctive TLS client fingerprint (e.g., Cobalt Strike's default JA3) even when the payload is fully encrypted.
Network forensics on a host with unknown malware: capturing JA3 hashes from outbound TLS and comparing against known-bad fingerprints to attribute the malware family.
T1071.001T1573
Thread 6
AMSI
PowerShell ScriptBlock Log Monitor
Tails Windows Event Log for Event ID 4104 (PowerShell ScriptBlock logging). Scores each script against 16 obfuscation patterns. Fires alerts at ≥2 matches. Requires ScriptBlock logging enabled in registry.
Automatically detecting a PowerShell download cradle (IEX + WebClient) the moment it executes, before the payload lands on disk, satisfying a "fileless attack" detection requirement.
Monitoring a developer workstation where PowerShell is frequently used — alerting only when obfuscation indicators cross the threshold, avoiding noise from legitimate scripts.
T1059.001T1027T1562.001
Thread 7
Correlator
Attack Chain Correlator (60s sweep)
Sweeps the event_timeline every 60 seconds for sequences matching 7 predefined multi-stage attack chains within a 10-minute window. On a match: writes to chain_alerts, fires a rich webhook embed with MITRE links and L1 triage steps.
Automatically escalating what appear to be separate low-severity events (a LSASS access + credential API call) into a single HIGH chain alert when they occur within 10 minutes of each other.
Reducing mean-time-to-detect for sophisticated multi-stage attacks where no single event would trigger a priority alert, but the sequence is unmistakably malicious.

SIEM Export
Telemetry Exporter — siem_export.py
Incremental Splunk HEC push and/or JSON Lines file export. Exports detections and behavioral telemetry. Re-runs are safe — only new records since the last export are included.
python siem_export.py [--mode file|hec|both] [--watch N]
--file
JSON Lines File Export
Exports all detections and event_timeline telemetry to a JSONL file since the last saved export state. Incremental — re-running appends only new records. Suitable for log shippers, Elastic, or offline analysis.
Scenarios
Daily scheduled task exporting the previous day's detections to a JSONL file that a Logstash pipeline picks up for ingestion into an on-premises Elastic SIEM.
End-of-engagement export: generating a JSONL archive of all detections across a two-week IR engagement for post-incident reporting and evidentiary archival.
Offline analysis: exporting telemetry to JSONL for import into a Jupyter notebook for statistical analysis of detection patterns across a fleet.
--hec
Splunk HEC Push
Pushes detections directly to a Splunk HTTP Event Collector endpoint with proper sourcetype tagging (cybersentinel:detection, cybersentinel:telemetry). Incremental state tracking prevents duplicate events.
Scenarios
Integrating CyberSentinel as a data source into an enterprise Splunk SIEM so that chain alerts and C2 detections appear in existing SOC dashboards and correlation rules.
One-shot historical import: using --reset to push the full detection history into a newly provisioned Splunk index during initial integration.
--watch
Continuous Watch Mode
Runs the export loop on a configurable interval (e.g., every 30 seconds), pushing incremental deltas to Splunk HEC or the JSONL file continuously. Keeps SIEM data near-real-time without a persistent agent.
Scenarios
Running as a Windows service alongside the daemon so that all detections appear in Splunk within 30 seconds for SOC alert correlation.
During an active incident, starting watch mode to ensure the SIEM receives a live feed of all new detections as fast as possible without manual re-export.
--both
Simultaneous File + HEC Export
Writes to both a JSONL archive and the Splunk HEC endpoint in a single run. The JSONL provides a local backup; the HEC push feeds the live SIEM. Configuration read from siem_config.json.
Scenarios
Regulated environments where a local audit trail is required in addition to SIEM forwarding — satisfying both data retention and real-time alerting requirements in one command.
Fallback resilience: if the Splunk HEC endpoint is temporarily unavailable, the JSONL file captures all missed events for replay once connectivity is restored.
Eval Harness
Benchmark Pipeline — eval_harness.py
Quantitative ML benchmark pipeline. Accepts labeled malware/clean sample directories. Produces precision, recall, F1, FPR, FNR, and accuracy across a threshold sweep. Results persist across restarts in v2_predictions.db.
python eval_harness.py  |  python CyberSentinel.py --evaluate
Mode A
Temporal Stratification Benchmark
Splits samples into pre-2020 and post-2020 subsets and evaluates detection performance on each independently. Reveals whether the model degrades on newer malware (concept drift indicator) or maintains consistent accuracy across time periods.
Scenarios
Model health check: comparing pre-2020 vs. post-2020 F1 scores to detect whether the model is over-fitted to historical malware and under-performing on current threats.
Research evaluation: publishing a paper on EMBER2024 model performance that requires temporal stratification to demonstrate generalization to new threat families.
Retraining validation: after adding post-2020 samples to training data, re-running the benchmark to confirm the post-2020 F1 improved without regressing on the pre-2020 set.
Mode B
Adversarial / UPX Stealth Dataset
Evaluates model performance against UPX-packed malware that deliberately obscures PE features. Uses a separate stealth/ subdirectory. Tests whether the LightGBM model can detect packers that reduce feature fidelity.
Scenarios
Red team evaluation: testing whether the model would catch their UPX-packed custom implant before deploying it in a purple team exercise.
Feature engineering research: identifying which EMBER2024 feature groups remain discriminative for packed executables to guide future feature work.
Adversarial robustness reporting for a security conference — quantifying the detection rate against common evasion techniques to contextualise claims about model effectiveness.
Mode C
Threshold Sweep (θ 0.40–0.80)
Runs the full sample set at each classification threshold from 0.40 to 0.80 in 0.05 steps, producing a confusion matrix and full metric set at each point. Helps choose the operational threshold that balances FPR vs. FNR for the deployment context.
Scenarios
A SOC team with high alert fatigue needs to raise the threshold to reduce FP volume, using the sweep to find the highest θ that still maintains an acceptable FNR.
High-security deployment (e.g., critical infrastructure) requires minimising FNR regardless of FP cost — the sweep identifies the lowest safe threshold.
Generating a ROC-style threshold table for inclusion in a model evaluation report, showing the full precision-recall tradeoff curve.
--tier1
Cloud Consensus Evaluation
Optional flag that routes each sample through the cloud API tier (VT, OTX, MetaDefender, MalwareBazaar) in addition to the ML tier, allowing a direct comparison of cloud consensus vs. offline ML accuracy on the same labeled sample set.
Scenarios
Justifying the need for offline ML: demonstrating that the local model catches malware that the cloud APIs miss (e.g., novel/private malware not yet in cloud databases).
API cost analysis: determining what proportion of cloud API calls are actually necessary by comparing cases where ML-only would have been sufficient.
Academic benchmarking: producing a fair comparison between CyberSentinel's tiered approach and pure cloud-lookup methods on the same dataset.
Output
JSON + TXT Forensic Report
After the pipeline completes, exports a JSON metrics file (machine-readable) and a human-readable TXT report containing the full threshold sweep table, per-sample raw score log, and summary statistics. Persists in v2_predictions.db across runs.
Scenarios
Thesis submission: generating the benchmark report that forms the quantitative results section of a cybersecurity research paper.
Enterprise deployment sign-off: sharing the TXT report with a security review board as evidence that the model meets the organisation's stated detection requirements.
Regression tracking: running the harness after each model update and diff-ing the JSON output against the previous version's report to catch any performance regressions.
